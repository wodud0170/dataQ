# 데이터 구조 진단 설계서

**작성일**: 2026-03-31
**상태**: 설계 (미구현)

---

## 1. 개요

### 1.1 목적
운영 중인 DBMS의 현재 상태를 재수집하여, 이전 수집 결과와의 변경사항(diff)을 파악하는 기능.
구조 진단은 **"DB가 어떻게 바뀌었는가"만 책임**지며, 표준 준수 여부 판단은 기존 표준 진단의 역할로 남긴다.

### 1.2 기존 기능과의 관계

| 기능 | 역할 | 범위 |
|------|------|------|
| **데이터 모델 수집** (기존) | DBMS 카탈로그 → TB_DATA_MODEL_ATTR 저장 | 수집만 |
| **구조 진단** (신규) | 재수집 → 이전 스냅샷과 diff | 변경 감지만 |
| **표준화 진단** (기존) | 수집된 컬럼 vs TB_TERMS 비교 | 표준 준수 검증 |

각 기능은 독립적으로 실행 가능하며, 선후관계를 강제하지 않는다.

### 1.3 메뉴 구조

```
데이터 표준화 진단
  ├─ 진단 실행          (기존 - 표준 진단)
  ├─ 진단 결과          (기존 - 표준 진단)
  └─ 구조 진단          (신규) ← 이 문서의 대상
```

## 2. 기능 상세

### 2.1 DBMS 재수집

**동작**: 기존 q-executor의 수집 기능을 호출하여 대상 데이터소스의 현재 DBMS 카탈로그를 다시 읽어옴.

```
[재수집 & 비교 실행]
    │
    ├─ 1. 현재 TB_DATA_MODEL_ATTR 데이터를 스냅샷 (PREV)
    │
    ├─ 2. q-executor 수집 API 호출 → TB_DATA_MODEL_ATTR 갱신 (CURR)
    │
    ├─ 3. PREV vs CURR diff 계산
    │
    ├─ 4. 결과를 TB_STRUCT_DIAG_* 테이블에 저장
    │
    └─ 5. 화면에 결과 표시
```

**대상 DBMS**: 현재 등록된 데이터소스 목록에서 선택
**수집 범위**: 스키마 단위 (전체 또는 선택)

### 2.2 변경 감지 (Diff)

이전 수집 시점과 현재 수집 결과를 비교하여 변경사항을 분류.

#### 변경 유형

| 유형 | 설명 | 표시색 |
|------|------|--------|
| **ADDED** | 이전에 없던 테이블/컬럼이 신규 생성됨 | 초록 |
| **MODIFIED** | 기존 컬럼의 데이터타입, 길이, Nullable 등이 변경됨 | 주황 |
| **DELETED** | 이전에 있던 테이블/컬럼이 삭제됨 | 빨강 |
| **UNCHANGED** | 변경 없음 | (목록에 표시하지 않음) |

#### diff 저장 구조

```sql
-- 구조 진단 이력
TB_STRUCT_DIAG_HISTORY
├─ DIAG_ID            VARCHAR(40)   PK   -- 진단 ID
├─ DS_ID              VARCHAR(40)        -- 데이터소스 ID
├─ SCHEMA_NM          VARCHAR(100)       -- 스키마명
├─ DIAG_DT            TIMESTAMP          -- 진단 일시
├─ PREV_COLLECT_DT    TIMESTAMP          -- 이전 수집 일시
├─ TOTAL_TABLES       INT                -- 전체 테이블 수
├─ TOTAL_COLUMNS      INT                -- 전체 컬럼 수
├─ ADDED_TABLES       INT                -- 추가된 테이블 수
├─ ADDED_COLUMNS      INT                -- 추가된 컬럼 수
├─ MODIFIED_COLUMNS   INT                -- 변경된 컬럼 수
├─ DELETED_TABLES     INT                -- 삭제된 테이블 수
├─ DELETED_COLUMNS    INT                -- 삭제된 컬럼 수
└─ CRET_USER_ID       VARCHAR(40)        -- 실행자

-- 구조 진단 상세
TB_STRUCT_DIAG_DETAIL
├─ DIAG_ID            VARCHAR(40)   PK   -- 진단 ID
├─ SEQ                INT           PK   -- 순번
├─ TABLE_NM           VARCHAR(200)       -- 테이블명
├─ COLUMN_NM          VARCHAR(200)       -- 컬럼명 (테이블 단위 변경은 NULL)
├─ CHANGE_TYPE        VARCHAR(20)        -- ADDED / MODIFIED / DELETED
├─ PREV_DATA_TYPE     VARCHAR(50)        -- 이전 데이터타입
├─ CURR_DATA_TYPE     VARCHAR(50)        -- 현재 데이터타입
├─ PREV_DATA_LEN      INT                -- 이전 길이
├─ CURR_DATA_LEN      INT                -- 현재 길이
├─ PREV_NULLABLE      CHAR(1)            -- 이전 Nullable
└─ CURR_NULLABLE      CHAR(1)            -- 현재 Nullable
```

## 3. 화면 설계

```
┌──────────────────────────────────────────────────────────┐

├──────────────────────────────────────────────────────────┤
│                                                          │
│  [데이터소스 선택 ▼]  [스키마 선택 ▼]  [재수집 & 비교 실행]   │
│                                                          │
│  ┌────────────┬────────────┬────────────┬──────────────┐ │
│  │  전체 테이블  │  전체 컬럼   │  변경 항목    │  마지막 수집  │ │

│  ┌─ 변경 요약 ──────────────────────────────────────┐     │
│  │  추가(ADDED): 테이블 2, 컬럼 8                    │     │
│  │  변경(MODIFIED): 컬럼 12                          │     │

```

### 3.2 변경사항 테이블 (메인)

| 테이블명 | 컬럼명 | 변경유형 | 이전 타입/길이 | 현재 타입/길이 | 비고 |
|---------|--------|---------|-------------|-------------|------|
| TB_USER | EMAIL_ADDR | ADDED | - | VARCHAR(200) | 신규 컬럼 |
| TB_ORDER | ORDER_AMT | MODIFIED | NUMBER(10) | NUMBER(15) | 길이 변경 |
| TB_ORDER | STATUS_CD | MODIFIED | CHAR(1) | VARCHAR(10) | 타입+길이 변경 |
| TB_LOG | OLD_FLAG | DELETED | CHAR(1) | - | 컬럼 삭제 |
| TB_NEW_TBL | (전체) | ADDED | - | 5 columns | 테이블 신규 |

- 변경유형별 필터 (칩 버튼)
- 테이블명 정렬
- 테이블 단위 ADDED/DELETED는 컬럼명에 "(전체)" 표시, 클릭 시 컬럼 목록 펼침

### 3.3 이전 진단 이력

과거 구조 진단 결과를 드롭다운으로 선택하여 조회 가능.

| 진단 일시 | 스키마 | 변경 건수 | 실행자 |
|----------|--------|---------|--------|
| 2026-03-30 14:30 | quality | 23건 | admin |
| 2026-03-25 10:00 | quality | 5건 | admin |
| 2026-03-15 09:00 | quality | 0건 (변경 없음) | space |

## 4. API 설계

| Method | URL | 설명 |
|--------|-----|------|
| POST | `/api/std/structDiag/execute` | 재수집 + diff 실행 |
| GET | `/api/std/structDiag/history` | 진단 이력 목록 |
| GET | `/api/std/structDiag/result/{diagId}` | 특정 진단 결과 (요약 + 상세) |
| GET | `/api/std/structDiag/result/{diagId}/export` | Excel 다운로드 |

### 실행 흐름

```
POST /structDiag/execute
  body: { dsId, schemaNm }
    │
    ├─ 1. TB_DATA_MODEL_ATTR에서 해당 스키마의 현재 데이터 조회 → PREV
    │
    ├─ 2. q-executor 수집 API 호출 → TB_DATA_MODEL_ATTR 갱신
    │
    ├─ 3. 갱신된 TB_DATA_MODEL_ATTR 조회 → CURR
    │
    ├─ 4. PREV vs CURR diff 계산
    │     ├─ PREV에만 있음 (table+column 기준) → DELETED
    │     ├─ CURR에만 있음 → ADDED
    │     └─ 둘 다 있지만 타입/길이/nullable 다름 → MODIFIED
    │
    ├─ 5. TB_STRUCT_DIAG_HISTORY에 요약 저장
    │
    ├─ 6. TB_STRUCT_DIAG_DETAIL에 변경 건만 저장 (UNCHANGED 제외)
    │
    └─ 7. 결과 반환
```

## 5. 기존 코드 활용 계획

| 구현 항목 | 기존 자산 | 추가 개발 |
|----------|----------|----------|
| DBMS 수집 | q-executor 수집 로직 그대로 호출 | 없음 |
| 이전 데이터 스냅샷 | TB_DATA_MODEL_ATTR SELECT | 수집 전 조회 로직 |
| diff 계산 | 없음 | Java에서 Map 비교 (table+column → 속성 비교) |
| 결과 저장 | 없음 | DDL 2개 + MyBatis 매퍼 |
| 화면 | DSDataDiag.vue 레이아웃 참고 | 신규 Vue 컴포넌트 (DSStructDiag.vue) |
| Excel 다운로드 | ExcelDownloadService 재사용 | 시트 구성 정의 |

## 6. 표준 진단과의 연계 (느슨한 연계)

구조 진단과 표준 진단은 독립적이지만, 표준 진단 화면에서 **구조 진단 상태를 뱃지로 표시**하여 데이터 신뢰도를 안내한다.

### 6.1 표준 진단 화면의 데이터소스/스키마 선택 영역

```
[데이터소스 선택 ▼]  [스키마 선택 ▼]

  quality 스키마  ✅ 구조 검증됨 (2026-03-30 14:30)    ← 초록 뱃지
  hr 스키마       ⚠ 구조 검증 3일 경과 (2026-03-27)    ← 주황 뱃지
  legacy 스키마   (구조 미검증)                         ← 뱃지 없음
```

### 6.2 뱃지 판정 기준

| 상태 | 조건 | 표시 |
|------|------|------|
| **검증됨** (초록) | 가장 최근 구조 진단에서 변경사항 0건 + 7일 이내 | ✅ 구조 검증됨 (일시) |
| **경과** (주황) | 구조 진단 이력은 있지만 7일 초과 | ⚠ 구조 검증 N일 경과 |
| **변경 감지** (빨강) | 가장 최근 구조 진단에서 변경사항 있음 | ⛔ 구조 변경 감지 (N건) |
| **미검증** (회색) | 구조 진단 이력 없음 | (표시 없음) |

### 6.3 구현 방식

- 표준 진단 화면 로드 시 `TB_STRUCT_DIAG_HISTORY`에서 해당 스키마의 최신 이력 1건 조회
- 변경건수 + 진단일시로 뱃지 판정
- 추가 API: `GET /api/std/structDiag/status?dsId=&schemaNm=` → `{ lastDiagDt, changeCount, daysSince }`

## 7. 구현 우선순위

| 순서 | 항목 | 필수 여부 |
|------|------|----------|
| 1 | DDL (TB_STRUCT_DIAG_HISTORY, DETAIL) | 필수 |
| 2 | 재수집 + diff 계산 백엔드 API | 필수 |
| 3 | 구조 진단 화면 (변경사항 테이블 + 필터 + 스탯 카드) | 필수 |
| 4 | 진단 이력 조회 | 권장 |
| 5 | Excel 리포트 다운로드 | 권장 |
| 6 | 표준 진단 화면에 구조 검증 뱃지 표시 | 권장 |

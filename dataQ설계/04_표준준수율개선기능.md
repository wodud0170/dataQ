# 표준 준수율 개선 기능 설계안

> 작성일: 2026-03-24

## 배경

현재 진단 결과 화면에서 이슈를 **확인**할 수는 있지만, 이를 **해결**하려면 용어/도메인/단어 등록 화면으로 직접 이동하거나, DB에서 컬럼을 수동으로 수정해야 한다. 진단 → 확인 → 해결의 사이클을 하나의 화면에서 자연스럽게 이어갈 수 있도록 개선한다.

## 현재 이슈 유형과 해결 경로

| 이슈 유형       | 원인                        | 현재 해결 방법               |
| ----------- | ------------------------- | ---------------------- |
| **용어 미존재**  | 컬럼 영문명에 매칭되는 표준 용어가 없음    | 용어 등록 화면 이동 → 수동 등록    |
| **한글명 불일치** | 컬럼 COMMENT와 표준 용어 한글명이 다름 | 용어 수정 또는 DB COMMENT 변경 |
| **타입 불일치**  | 컬럼 타입 ≠ 도메인 표준 타입         | 도메인 수정 또는 ALTER TABLE  |
| **길이 불일치**  | 컬럼 길이 ≠ 도메인 표준 길이         | 도메인 수정 또는 ALTER TABLE  |

**문제**: 모든 해결 방법이 화면 이동 + 수동 작업을 요구하며, 이슈가 수백 건일 경우 반복 작업이 과도함.

## 기능 제안

### 1. 용어 빠른 등록 (Quick Term Registration)

> 이슈 유형: **용어 미존재 (TERM_NOT_EXIST)**

#### 개념
진단 결과 컬럼 상세 탭에서 "용어 미존재" 이슈가 있는 행에 **[용어 등록]** 버튼을 표시.
클릭 시 용어 등록 모달이 열리며, 해당 컬럼 정보가 자동으로 채워짐.

#### 자동 채움 정보
| 용어 등록 필드 | 자동 채움 값 | 출처 |
|-------------|-----------|------|
| 용어명 (termsNm) | 컬럼 한글명 | attrNmKr (COMMENT) |
| 영문 약어명 (termEngAbrvNm) | 컬럼 영문명 | attrNm |
| 도메인 (domainNm) | 타입+길이 기반 추천 | actualDataType + actualDataLen |
| 설명 (termDesc) | "테이블명.컬럼명에서 수집" | 자동 생성 |

#### 문제: 등록 체인 의존성

용어를 등록하려면 **단어**가 먼저 있어야 하고, 단어에는 **도메인 분류**가 필요하다.
현재 구조에서 용어 하나를 등록하기 위한 최악의 경로:

```
용어 등록하려는데 → 단어가 없네 → 단어 등록 화면 이동
  → 단어 등록 → 다시 용어 등록 화면으로 복귀
  → 도메인도 없네 → 도메인 등록 화면 이동
  → 도메인 등록 → 다시 용어 등록 화면으로 복귀
  → 드디어 용어 등록 완료 (화면 이동 4번)
```

이슈가 수백 건이면 사실상 사용 불가능.

#### 해결: 원스텝 통합 등록

컬럼 영문명(예: `BASS_DT`)은 이미 단어 약어의 조합(`BASS` + `DT`)이므로,
언더스코어(`_`) 기준으로 분리하면 필요한 단어 목록을 자동으로 추출할 수 있다.

**통합 등록 흐름:**
```
컬럼 상세 탭 → "용어 미존재" 행 → [용어 등록] 버튼
  → 통합 등록 모달 오픈
  → Step 1: 자동 분석 결과 표시
    ┌──────────────────────────────────────────────┐
    │  컬럼명: BASS_DT                             │
    │  한글명: 기준일자                              │
    │                                              │
    │  [단어 분석]                                  │
    │  ┌──────┬────────┬────────┬────────┐         │
    │  │ 약어  │ 단어명  │ 상태    │ 액션    │         │
    │  ├──────┼────────┼────────┼────────┤         │
    │  │ BASS │ 기준   │ ✅ 등록됨│        │         │
    │  │ DT   │ 일자   │ ❌ 미등록│ [등록]  │         │
    │  └──────┴────────┴────────┴────────┘         │
    │                                              │
    │  [도메인]                                     │
    │  VARCHAR2(8) → 추천: "일자" 도메인  ✅ 존재    │
    │                                              │
    │  ※ 미등록 단어가 있습니다. 아래에서 바로 등록    │
    └──────────────────────────────────────────────┘

  → Step 2: 미등록 단어 인라인 등록 (화면 이동 없음)
    ┌──────────────────────────────────────────────┐
    │  [미등록 단어 등록]                            │
    │                                              │
    │  단어명: 일자          (한글명에서 자동 추출)    │
    │  영문약어: DT          (컬럼명에서 자동 채움)    │
    │  영문명:  Date         (자동 추천)             │
    │  설명:   날짜를 나타내는 단어                    │
    │                                              │
    │              [단어 등록]                       │
    └──────────────────────────────────────────────┘

  → Step 3: 모든 단어 준비 완료 → 용어 등록
    ┌──────────────────────────────────────────────┐
    │  [용어 등록]                                  │
    │                                              │
    │  용어명:    기준일자     (자동)                 │
    │  영문약어:  BASS_DT     (자동, readonly)       │
    │  구성단어:  기준(BASS) + 일자(DT)  ✅ 모두 존재 │
    │  도메인:    일자 (VARCHAR2, 8)                 │
    │  설명:     기준이 되는 일자                     │
    │                                              │
    │              [등록 완료]                       │
    └──────────────────────────────────────────────┘

  → 완료: 단어 + 용어가 한 모달에서 모두 등록됨
```

#### 핵심 자동화 로직

| 단계 | 자동화 내용 | 방법 |
|------|-----------|------|
| 단어 분리 | `BASS_DT` → `[BASS, DT]` | 언더스코어 split |
| 단어 존재 확인 | 각 약어가 TB_WORD에 있는지 | 약어 기준 조회 API |
| 한글명 분리 | `기준일자` → `[기준, 일자]` | 등록된 단어 한글명 매칭 순서대로 분리 |
| 도메인 추천 | 컬럼 타입+길이 → 가장 유사한 도메인 | 타입 일치 + 길이 일치 기준 정렬 |
| 미등록 단어 한글명 | 전체 한글명에서 기등록 단어를 제외한 나머지 | 순서 기반 추출 |

#### 확장: 일괄 등록

여러 컬럼을 선택하여 한번에 처리:

```
컬럼 상세 탭 → 여러 행 체크 → [선택 용어 일괄 등록]
  → 일괄 분석 결과 미리보기
    ┌────────────┬──────────┬───────────┬──────────┬────────┐
    │ 컬럼명      │ 한글명   │ 미등록 단어 │ 도메인    │ 상태    │
    ├────────────┼──────────┼───────────┼──────────┼────────┤
    │ BASS_DT    │ 기준일자  │ DT        │ 일자 ✅   │ 단어필요│
    │ REGIST_NO  │ 등록번호  │ (없음)     │ 번호 ✅   │ 바로등록│
    │ APRV_YN    │ 승인여부  │ APRV      │ 여부 ✅   │ 단어필요│
    │ XYZ_CD     │ 특수코드  │ XYZ       │ 코드 ✅   │ 단어필요│
    └────────────┴──────────┴───────────┴──────────┴────────┘

    미등록 단어 3건을 먼저 등록합니다:
    ┌──────┬────────┬────────┐
    │ 약어  │ 한글명  │ 영문명  │
    ├──────┼────────┼────────┤
    │ DT   │ 일자   │ Date   │  ← 수정 가능
    │ APRV │ 승인   │ Approve│  ← 수정 가능
    │ XYZ  │ 특수   │        │  ← 직접 입력
    └──────┴────────┴────────┘

    [단어 일괄 등록] → [용어 일괄 등록] → 완료
```

### 2. COMMENT ON 스크립트 생성

> 이슈 유형: **한글명 불일치 (TERM_KR_NM_MISMATCH)**

#### 개념
한글명 불일치 이슈를 DB 측에서 해결할 때, 표준 한글명으로 COMMENT를 수정하는 SQL 스크립트를 자동 생성하여 다운로드.

#### 생성 예시
```sql
-- 테이블: TB_CNPT_BASE
COMMENT ON COLUMN TB_CNPT_BASE.BASS_DT IS '기준일자';
COMMENT ON COLUMN TB_CNPT_BASE.REGIST_DT IS '등록일시';

-- 테이블: TB_BLDG_INFO
COMMENT ON COLUMN TB_BLDG_INFO.BLDG_NM IS '건물명';
```

#### 동작 흐름
```
진단 결과 탭 또는 컬럼 상세 탭
  → [COMMENT 스크립트 생성] 버튼
  → 대상: 한글명 불일치 컬럼 전체 (또는 선택된 행만)
  → DBMS 타입 선택 (Oracle/PostgreSQL/MySQL 등)
  → 스크립트 미리보기 → 다운로드 (.sql)
```

#### DBMS별 문법
| DBMS | 문법 |
|------|------|
| Oracle | `COMMENT ON COLUMN 스키마.테이블.컬럼 IS '한글명';` |
| PostgreSQL | `COMMENT ON COLUMN 테이블.컬럼 IS '한글명';` |
| MySQL | `ALTER TABLE 테이블 MODIFY COLUMN 컬럼 타입 COMMENT '한글명';` |

### 3. ALTER TABLE 스크립트 생성

> 이슈 유형: **타입 불일치 (DATA_TYPE_MISMATCH)**, **길이 불일치 (DATA_LEN_MISMATCH)**

#### 개념
실제 컬럼의 타입/길이를 표준에 맞게 변경하는 ALTER TABLE 스크립트를 자동 생성.

#### 생성 예시
```sql
-- 타입 불일치 수정
ALTER TABLE TB_CNPT_BASE MODIFY REGIST_DT VARCHAR2(20);  -- CHAR(20) → VARCHAR2(20)

-- 길이 불일치 수정
ALTER TABLE TB_DLNG_HIST MODIFY DLNG_AMT NUMBER(15);     -- NUMBER(10) → NUMBER(15)
```

#### 주의사항
- 타입 변경은 데이터 손실 위험이 있으므로 **경고 문구** 포함
- 스크립트 상단에 `-- [주의] 실행 전 반드시 백업하세요` 안내
- 축소 변경(길이가 줄어드는 경우)은 별도 경고 표시

### 4. 표준 사전 측 수정 (역방향 대응)

> 모든 이슈 유형

#### 개념
이슈를 항상 "DB를 표준에 맞춘다"로만 해결하는 것이 아니라, 반대로 **표준 사전을 현실에 맞게 수정**하는 경로도 제공.

#### 동작
컬럼 상세 탭에서 이슈 행 클릭 시 두 가지 선택지:

```
┌─────────────────────────────────────┐
│  이 이슈를 어떻게 해결하시겠습니까?    │
│                                     │
│  [DB 수정] 컬럼을 표준에 맞춤        │
│    → ALTER/COMMENT 스크립트 생성     │
│                                     │
│  [표준 수정] 표준을 현재 DB에 맞춤    │
│    → 용어/도메인 수정 모달 오픈       │
│                                     │
│  [무시] 이 이슈를 무시 처리           │
│    → 사유 입력 후 제외 목록에 추가    │
└─────────────────────────────────────┘
```

### 5. 이슈 무시/예외 처리

#### 개념
일부 컬럼은 의도적으로 표준과 다를 수 있다 (레거시, 외부 연동 등).
이런 컬럼을 **예외 목록**에 등록하면 다음 진단 시 이슈에서 제외.

#### 필요 데이터
```
TB_DIAG_EXCEPTION (신규 테이블)
  EXCEPTION_ID    (PK)
  DM_ID           (데이터모델)
  OBJ_NM          (테이블명)
  ATTR_NM         (컬럼명, NULL이면 테이블 전체)
  DIAG_TYPE       (예외 처리할 이슈 유형, NULL이면 전체)
  REASON          (사유)
  CRET_USER_ID    (등록자)
  CRET_DT         (등록일)
```

#### 효과
- 진단 시 예외 목록 참조 → 해당 컬럼/이슈 스킵
- 진단 결과 탭에 "예외 처리 N건" 별도 표시
- 준수율 계산: 예외 건은 분모에서 제외 (또는 별도 옵션)

### 6. 재진단 원클릭

#### 개념
표준 사전 수정이나 DB 수정 후, 바로 재진단을 실행하여 준수율 변화를 확인.

#### 동작
```
진단 결과 탭 → [재진단] 버튼
  → 동일한 수집 데이터 기반으로 진단만 재실행
  → 완료 후 준수율 변화 표시: 72% → 85% (↑13%p)
```

#### 참고
- 재수집 없이 진단만 재실행 (빠름)
- COMMENT 변경 등 DB 변경이 있었으면 재수집 후 재진단 필요 → 안내 표시

## 우선순위 제안

| 순위  | 기능                      | 효과  | 난이도 | 비고                    |
| :-: | ----------------------- | --- | :-: | --------------------- |
|  1  | **COMMENT 스크립트 생성**     | 높음  | 낮음  | 프론트 단독 구현 가능, 즉시 활용   |
|  2  | **ALTER TABLE 스크립트 생성** | 높음  | 낮음  | COMMENT과 유사 구조, 함께 구현 |
|  3  | **용어 빠른 등록**            | 높음  | 중간  | 기존 용어 등록 모달 재활용       |
|  4  | **재진단 원클릭**             | 중간  | 낮음  | 기존 진단 API 재호출         |
|  5  | **이슈 무시/예외 처리**         | 중간  | 중간  | 신규 테이블 + 진단 로직 수정     |
|  6  | **표준 사전 측 수정**          | 중간  | 높음  | 용어/도메인 수정 연동 복잡       |

## 구현 시 화면 변화 (컬럼 상세 탭)

### Before (현재)
```
┌──────────┬──────────┬────────┬────────┬──────┬──────┬──────────┬──────┬──────┬──────────┐
│ 테이블명  │ 한글명   │ 컬럼명  │ 한글명  │ 타입  │ 길이  │ 표준한글명│표준타입│표준길이│ 진단결과  │
├──────────┼──────────┼────────┼────────┼──────┼──────┼──────────┼──────┼──────┼──────────┤
│ TB_TEST  │ 테스트   │ COL_A  │ 컬럼A  │ CHAR │ 20   │ 컬럼에이 │VARCHAR│ 20  │ 한글명   │
│          │          │        │        │      │      │          │       │     │ 타입     │
└──────────┴──────────┴────────┴────────┴──────┴──────┴──────────┴──────┴──────┴──────────┘
```

### After (개선 후)
```
┌──┬──────────┬──────────┬────────┬────────┬──────┬──────┬──────────┬──────┬──────┬──────────┬──────┐
│☐ │ 테이블명  │ 한글명   │ 컬럼명  │ 한글명  │ 타입  │ 길이  │ 표준한글명│표준타입│표준길이│ 진단결과  │ 액션  │
├──┼──────────┼──────────┼────────┼────────┼──────┼──────┼──────────┼──────┼──────┼──────────┼──────┤
│☐ │ TB_TEST  │ 테스트   │ COL_A  │ 컬럼A  │ CHAR │ 20   │ 컬럼에이 │VARCHAR│ 20  │ 한글명   │ [▼]  │
│  │          │          │        │        │      │      │          │       │     │ 타입     │      │
└──┴──────────┴──────────┴────────┴────────┴──────┴──────┴──────────┴──────┴──────┴──────────┴──────┘

 ☑ 2건 선택                    [COMMENT 스크립트] [ALTER 스크립트] [용어 일괄 등록] [예외 처리]
```

- **☐ 체크박스**: 행 선택 (일괄 작업용)
- **[▼] 액션 버튼**: 클릭 시 해당 이슈의 해결 옵션 드롭다운
- **하단 액션 바**: 선택된 행에 대한 일괄 작업 버튼

## 참고: 현재 시스템 제약

| 항목 | 현황 | 영향 |

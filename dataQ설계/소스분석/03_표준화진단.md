# 소스분석 - 데이터 표준화 진단

> 최종 수정: 2026-03-24

## 개요

수집된 데이터모델의 컬럼 정보와 표준 사전(용어·도메인)을 비교하여 표준화 위반 항목을 탐지하는 기능.
백그라운드 배치로 실행되며, 실행 중 폴링으로 진행률을 확인한다.

## 기능 구성

```
데이터 표준화 진단
├── 진단 실행 (DSDataDiag.vue)         JOB 생성·실행·중지·이력 조회
└── 진단 결과 (DSDataDiagResult.vue)   JOB별 결과 조회 (테이블 집계 / 컬럼 상세)
```

---

## 진단 실행 흐름

```
1. DSDataDiag.vue:
   - 데이터모델명 선택 (autocomplete → getDataModelStatsList)
   - 수집일시 선택 (select → getDataModelClctList)
   - "진단 시작" 클릭

2. POST /api/diag/startDiag { clctId, dataModelId, dataModelNm, clctDt }
   → q-center DiagController:
     a. TB_DIAG_JOB INSERT (status=READY)
     b. POST q-executor /api/diag/runDiag { diagJobId, clctId }
     c. Response.contents = diagJobId 반환

3. DSDataDiag.vue:
   - diagJobId 수신
   - setInterval 3초마다 GET /api/diag/getDiagJobById?diagJobId=... 폴링
   - 진행바 업데이트 (processCnt / totalCnt)
   - status 변화 감지:
     DONE    → 폴링 중단 + 완료 메시지
     STOPPED → 폴링 중단
     ERROR   → 폴링 중단 + 오류 메시지

4. q-executor DiagService (별도 스레드):
   (아래 진단 로직 참고)
```

## 진단 로직 (DiagService.java)

### 사전 준비
```
1. TB_DIAG_JOB status → RUNNING, startDt 기록
2. TB_DATA_MODEL_ATTR에서 수집 컬럼 전체 로드 (clctId 기준)
3. selectAllTermsForDiag로 전체 용어 로드
   → HashMap 구성:
     termsByEng : 영문명(TERMS_ENG_ABRV_NM) → StdTermsVo
     동의어(ALLOPH_SYNM_LST)도 Map에 추가
```

### 컬럼별 진단 규칙

영문명(attrNm) 기준으로 용어를 매칭하고, 순차적으로 검증한다.

```
컬럼 (attrNm=영문명, attrNmKr=한글명/COMMENTS) 에 대해:

STEP 1. 용어 존재 확인 (영문명 기준)
  termByEng = termsByEng.get(attrNm)
  if termByEng == null → TERM_NOT_EXIST (진단 종료, 다음 컬럼으로)

STEP 2. 한글명 일치 확인
  if attrNmKr이 존재하고 termByEng.termsNm과 다르면 → TERM_KR_NM_MISMATCH

STEP 3. 도메인 검증 (타입·길이)
  checkDomain(termByEng):
    domain = termByEng.domainNm → TB_DOMAIN 조회
    if domain != null:
      a. DATA_TYPE 비교 (case-insensitive)
         불일치 → DATA_TYPE_MISMATCH
      b. DATA_LEN 비교 (길이 또는 소수점 자릿수가 다르면)
         불일치 → DATA_LEN_MISMATCH
  ※ 타입과 길이가 둘 다 다르면 같은 컬럼에 2건(각각 별도)으로 기록
```

> **변경 사유 (기존 대비)**
> - 기존: 한글명/영문명 양방향 매칭 → `TERM_ENG_NM_MISMATCH` 유형 존재
> - 변경: 영문명 기준 단방향 매칭. 영문명이 용어에 없으면 바로 `TERM_NOT_EXIST`.
>   한글명만 있고 영문명이 없는 경우는 용어 미존재와 동일하므로 별도 유형 불필요.
> - `DATA_LEN_MISMATCH` 기준: 컬럼의 길이(DATA_LEN)와 소수점 자릿수(DATA_DECIMAL_LEN) 중 하나라도 표준과 다르면 불일치

### 진단 유형 정의
| 코드 | 설명 | stdValue | actualValue |
|------|------|---------|------------|
| `TERM_NOT_EXIST` | 영문명 기준 표준 용어 없음 | - | 컬럼 영문명 |
| `TERM_KR_NM_MISMATCH` | 용어는 있으나 한글명 불일치 | 표준 한글명 | 실제 한글명(COMMENTS) |
| `DATA_TYPE_MISMATCH` | 데이터 타입 표준 불일치 | 표준 타입 | 실제 타입 |
| `DATA_LEN_MISMATCH` | 데이터 길이 표준 불일치 (소수점 포함) | 표준 길이 | 실제 길이 |

> `TERM_ENG_NM_MISMATCH` 유형은 **폐지**. 영문명이 없으면 `TERM_NOT_EXIST`로 통합.

### JOB 상태 전이
```
READY
  └─ RUNNING (진단 시작)
        ├─ DONE (정상 완료)
        ├─ STOPPED (중지 요청 수신)
        │    └─ TB_DIAG_RESULT 삭제 (결과 초기화)
        └─ ERROR (예외 발생)
```

### 중지 처리
```java
// DiagService 내 정적 관리
static ConcurrentHashMap<String, AtomicBoolean> stopFlags;

// q-center /stopDiag 호출 → q-executor /api/diag/stopDiag
// → DiagService.requestStop(diagJobId) → stopFlags에 true 설정

// 진단 루프에서 주기적 체크
if (stopFlags.get(diagJobId).get()) {
    // TB_DIAG_RESULT 삭제
    // TB_DIAG_JOB status = STOPPED
    return;
}
```

- 50건 단위로 TB_DIAG_RESULT batch insert
- 처리 후 TB_DIAG_JOB.PROCESS_CNT, RESULT_CNT 업데이트

## 진단 결과 조회

### DSDataDiagResult.vue 구성

```
┌──────────────────────────────────────────────────────────────┐
│ [진단 이력 ▼] [테이블명] [컬럼명] [진단유형 ▼]                  │  ← 필터 바
├─────────────────────┬────────────────────────────────────────┤
│ 테이블 집계          │ 컬럼 상세                                │  ← 탭
└─────────────────────┴────────────────────────────────────────┘
```

**테이블 집계 탭** (selectDiagResultSummary)
```sql
SELECT OBJ_NM, COUNT(DISTINCT ATTR_NM) AS ISSUE_COL_CNT,
  COUNT(*) AS TOTAL_ISSUE_CNT,
  COUNT(*) FILTER (WHERE DIAG_TYPE = 'TERM_NOT_EXIST') AS TERM_NOT_EXIST_CNT,
  COUNT(*) FILTER (WHERE DIAG_TYPE = 'TERM_KR_NM_MISMATCH') AS TERM_KR_NM_MISMATCH_CNT,
  COUNT(*) FILTER (WHERE DIAG_TYPE = 'DATA_TYPE_MISMATCH') AS DATA_TYPE_MISMATCH_CNT,
  COUNT(*) FILTER (WHERE DIAG_TYPE = 'DATA_LEN_MISMATCH') AS DATA_LEN_MISMATCH_CNT
FROM TB_DIAG_RESULT
WHERE DIAG_JOB_ID = #{diagJobId}
GROUP BY OBJ_NM
```

### 컬럼 상세 탭 — 설계 변경

#### 변경 사유
- **기존**: 하나의 컬럼에 이슈가 여러 개이면 **여러 행**으로 표시 (예: 한글명 불일치 1행 + 길이 불일치 1행)
- **문제**: 사용자가 한 컬럼의 전체 상태를 파악하려면 여러 행을 대조해야 함
- **변경**: 컬럼 1건 = 1행. 표준 용어/도메인 정보를 JOIN하여 실제값과 표준값을 한눈에 비교

#### 컬럼 상세 화면 레이아웃
```
┌──────────┬──────────┬──────────┬────────┬────────┬──────────┬──────────┬──────────┬────────────┐
│ 테이블명  │ 컬럼영문명│ 컬럼한글명│ 타입    │ 길이    │ 표준한글명│ 표준타입  │ 표준길이  │ 진단결과    │
│ (실제)   │ (실제)    │ (실제)   │ (실제)  │ (실제)  │ (용어)   │ (도메인) │ (도메인)  │ (이슈유형)  │
├──────────┼──────────┼──────────┼────────┼────────┼──────────┼──────────┼──────────┼────────────┤
│TBL_MSG   │USER_ID   │ (없음)   │VARCHAR │ 16     │사용자ID  │ VARCHAR  │ 20       │한글명,길이  │
│TBL_MSG   │MSG_TYPE  │메시지유형 │VARCHAR │ 10     │메시지유형│ VARCHAR  │ 10       │ -          │
│TBL_MSG   │UNKNOWN   │알수없음   │VARCHAR │ 50     │   -      │    -     │    -     │용어미존재   │
└──────────┴──────────┴──────────┴────────┴────────┴──────────┴──────────┴──────────┴────────────┘
```

- **실제값**: TB_DIAG_RESULT의 OBJ_NM, ATTR_NM, ATTR_NM_KR + 수집 컬럼의 타입/길이
- **표준값**: 용어(TB_STND_TERMS) JOIN → 한글명(TERMS_NM), 도메인(TB_STND_DOMAIN) JOIN → 타입/길이
- **진단결과**: 해당 컬럼의 이슈 유형을 칩(chip)으로 표시 (복수 가능)
- **용어 미존재** 컬럼은 표준값 열이 전부 "-"로 표시

#### 조회 쿼리 (selectDiagResultDetail)

TB_DIAG_RESULT를 컬럼 단위로 GROUP BY하고, 용어/도메인을 LEFT JOIN하여 한 행에 표준+실제+이슈를 모두 담는다.

```sql
SELECT
    R.OBJ_NM,
    R.ATTR_NM,
    R.ATTR_NM_KR,
    -- 실제 컬럼 타입/길이 (수집 데이터에서 가져옴)
    A.DATA_TYPE       AS ACTUAL_DATA_TYPE,
    A.DATA_LEN        AS ACTUAL_DATA_LEN,
    -- 표준 용어/도메인 정보
    T.TERMS_NM        AS STD_TERMS_NM,
    D.DATA_TYPE       AS STD_DATA_TYPE,
    D.DATA_LEN        AS STD_DATA_LEN,
    -- 이슈 유형 집계 (컬럼 1행에 여러 이슈를 콤마로 연결)
    STRING_AGG(DISTINCT R.DIAG_TYPE, ',') AS DIAG_TYPES
FROM TB_DIAG_RESULT R
-- 수집 컬럼의 실제 타입/길이 조회
LEFT JOIN TB_DATA_MODEL_ATTR A
    ON A.DM_CLCT_ID = (SELECT DM_CLCT_ID FROM TB_DIAG_JOB WHERE DIAG_JOB_ID = #{diagJobId})
    AND A.ATTR_NM = R.ATTR_NM
    AND A.OBJ_NM = R.OBJ_NM
-- 표준 용어 조회 (영문명 기준)
LEFT JOIN TB_STND_TERMS T
    ON T.TERMS_ENG_ABRV_NM = R.ATTR_NM
-- 표준 도메인 조회 (용어의 도메인)
LEFT JOIN TB_STND_DOMAIN D
    ON D.DOMAIN_NM = T.DOMAIN_NM
WHERE R.DIAG_JOB_ID = #{diagJobId}
GROUP BY R.OBJ_NM, R.ATTR_NM, R.ATTR_NM_KR,
         A.DATA_TYPE, A.DATA_LEN,
         T.TERMS_NM, D.DATA_TYPE, D.DATA_LEN
ORDER BY R.OBJ_NM, R.ATTR_NM
```

> **참고**: 위 쿼리의 실제 테이블명/컬럼명은 기존 MyBatis mapper 기준으로 조정 필요.
> TB_DATA_MODEL_ATTR의 정확한 컬럼명은 datamodel.xml의 selectDataModelAttrListByClctId 참고.

#### TB_DIAG_RESULT 변경 여부

TB_DIAG_RESULT 테이블 구조는 **변경하지 않는다**.
- 진단 로직(DiagService)은 기존과 동일하게 이슈별로 개별 INSERT
- 조회 시점에 GROUP BY + JOIN으로 컬럼 단위 1행 표시
- 이유: 개별 이슈 이력 보존, 진단 로직 변경 최소화

#### 화면 컬럼 정의 (컬럼 상세 탭)

| 헤더 | 데이터 | 설명 |
|------|--------|------|
| 테이블명 | objNm | 실제 테이블명 |
| 컬럼 영문명 | attrNm | 실제 컬럼 영문명 |
| 컬럼 한글명 | attrNmKr | 실제 컬럼 COMMENTS (없으면 빈칸) |
| 실제 타입 | actualDataType | 수집된 컬럼의 데이터 타입 |
| 실제 길이 | actualDataLen | 수집된 컬럼의 데이터 길이 |
| 표준 한글명 | stdTermsNm | 용어 테이블의 한글명 (용어 미존재 시 "-") |
| 표준 타입 | stdDataType | 도메인 테이블의 데이터 타입 (용어 미존재 시 "-") |
| 표준 길이 | stdDataLen | 도메인 테이블의 데이터 길이 (용어 미존재 시 "-") |
| 진단결과 | diagTypes | 이슈 유형 칩 (복수 표시, 예: `한글명 불일치` `길이 불일치`) |

- 클라이언트 필터: 테이블명, 컬럼명, 진단유형 (대소문자 무시)
- 필터 변경 시 페이징 리셋 (`:key` 기반 리마운트)

## 현재 한계 / 개선 여지

| 항목      | 현황                   | 개선 방향                                |
| ------- | -------------------- | ------------------------------------ |
| 진단 범위   | 컬럼 한글명·영문명·타입·길이만 검증 | NULL 허용 여부, 기본값, FK 관계 등 확장          |
| 코드값 검증  | 미구현                  | 도메인 ALLOW_VAL_LST vs 실제 데이터 샘플 비교    |
| 결과 내보내기 | 화면 조회만 가능            | Excel 다운로드 기능 추가                     |
| 진단 비교   | 단일 수집이력 기준만 가능       | 두 수집이력 간 비교(diff) 기능                 |
| 진단 스케줄링 | 수동 실행만 가능            | 수집 완료 후 자동 진단 트리거                    |
| 결과 히스토리 | STOPPED JOB은 결과 삭제   | STOPPED 이력은 남기고 결과만 삭제하는 방식 고려       |
| 성능      | 전체 용어를 메모리에 로드       | 용어 수가 매우 많을 경우 OOM 위험 → 페이징 또는 캐시 도입 |

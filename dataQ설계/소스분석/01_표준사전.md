# 소스분석 - 데이터 표준 사전

> 최종 수정: 2026-03-23

## 개요

용어, 단어, 도메인, 코드 4가지 표준 객체를 관리하는 기능.
데이터모델 수집 후 표준화 진단의 기준 데이터로 활용된다.

## 기능 구성

```
표준 사전
├── 용어 (DSTerm.vue)          TB_TERMS
├── 단어 (DSWord.vue)          TB_WORD
├── 도메인 (DSDomain.vue)      TB_DOMAIN
│   ├── 도메인 그룹 (DSDomainGroup.vue)
│   └── 도메인 분류 (DSDomainClassification.vue)
└── 코드 (DSCode.vue)          TB_CODE_INFO + TB_CODE_DATA
```

## 용어 (TB_TERMS)

### 역할
표준화 진단의 핵심 기준. 컬럼의 한글명/영문명이 용어 사전에 있는지 여부로 표준화 수준 판단.

### 주요 필드
| 필드 | 설명 |
|------|------|
| TERMS_NM | 한글 용어명 (진단 시 컬럼 한글명과 비교) |
| TERMS_ENG_ABRV_NM | 영문 약어명 (진단 시 컬럼 영문명과 비교) |
| DOMAIN_NM | 연결된 도메인 (→ 데이터 타입/길이 검증에 사용) |
| ALLOPH_SYNM_LST | 동의어/이음이의어 목록 |
| TOKEN_LST | KOMORAN 형태소 분석 결과 (자동화 매핑용) |

### 핵심 쿼리
```sql
-- 진단용 전체 용어 로드 (도메인 JOIN)
SELECT T.TERMS_ID, T.TERMS_NM, T.TERMS_ENG_ABRV_NM, T.DOMAIN_NM,
       T.ALLOPH_SYNM_LST, D.DATA_TYPE, D.DATA_LEN, D.DATA_DECIMAL_LEN
FROM TB_TERMS T LEFT JOIN TB_DOMAIN D ON T.DOMAIN_NM = D.DOMAIN_NM
-- (selectAllTermsForDiag in terms.xml)
```

### CRUD 흐름
```
화면 입력 → DSTerm.vue
  → POST /api/std/createTerms (또는 update, delete)
  → q-center DataStandardController
  → MyBatis termsMapper
  → TB_TERMS
```

### 일괄등록 (Excel 업로드) 흐름
```
DSTerm.vue: 파일 선택 → POST /api/std/uploadTermsList (multipart)
  → q-center: 파일 수신 → POST q-executor /api/std/uploadTermsList (파일 전달)
  → q-executor: runService(DataStandardService) [동기 실행]
    → ExcelUploadService로 파싱
    → 기존 데이터 삭제 후 batch insert
    → HTTP 응답 반환
  → q-center: 응답 받아 Vue에 전달
  → DSTerm.vue: .then()에서 로딩 해제 + 목록 새로고침
```

> **이슈 이력**: 이전에는 WebSocket 완료 메시지로 처리했으나 장시간 처리 후 메시지 유실 문제 발생.
> `runService` (동기) 방식으로 변경하여 HTTP 응답을 완료 신호로 사용.

## 단어 (TB_WORD)

### 역할
용어를 구성하는 단위 단어 관리. 용어 토큰화 결과와 매핑하여 단어 표준화율 산출에 활용.

### 주요 필드
| 필드 | 설명 |
|------|------|
| WORD_NM | 단어명 (한글) |
| WORD_ENG_NM | 영문명 |
| WORD_MEANING | 의미 |
| WORD_CATEGORY | 범주 |

### 특이사항
- 단어 자체보다는 용어 토큰 분석의 참조 데이터 역할
- KOMORAN으로 용어를 형태소 분석 → 결과 토큰을 TB_WORD와 매핑

---

## 도메인 (TB_DOMAIN)

### 역할
컬럼의 데이터 타입/길이 표준 정의. 진단 시 도메인 기준으로 DATA_TYPE, DATA_LEN 검증.

### 주요 필드
| 필드 | 설명 |
|------|------|
| DOMAIN_NM | 도메인명 (용어의 DOMAIN_NM FK) |
| DATA_TYPE | 표준 데이터 타입 |
| DATA_LEN | 표준 길이 |
| DATA_DECIMAL_LEN | 소수 자릿수 |
| ALLOW_VAL_LST | 허용값 목록 |
| STOR_FMT | 저장 형식 |
| EXPR_FMT_LST | 표현 형식 목록 |
| APRV_YN | 승인 여부 |

### 도메인 그룹 / 도메인 분류
- 별도 테이블 없이 TB_DOMAIN의 DOMAIN_GRP_NM, DOMAIN_CLSF_NM 컬럼으로 관리
- GROUP BY로 집계하여 선택 목록 제공

### 진단 연계
```
TB_TERMS.DOMAIN_NM → TB_DOMAIN.DOMAIN_NM
  → DATA_TYPE 비교 (case-insensitive)
  → DATA_LEN 비교 (컬럼 길이 > 표준 → DATA_LEN_MISMATCH)
```

## 코드 (TB_CODE_INFO + TB_CODE_DATA)

### 역할
공통 코드 관리. 허용값 목록이 있는 도메인에서 실제 코드값 검증에 활용 가능(현재는 진단 미연계).

### 구조
```
TB_CODE_INFO (코드 그룹)
  └── TB_CODE_DATA (개별 코드값)
       CODE_ID FK 로 연결
```

---

## 승인 연계

표준 사전 객체(도메인, 용어, 단어)는 `APRV_YN` 필드로 승인 상태 관리.
`TB_APPROVE_STAT` 테이블에 이력 저장. 자세한 내용은 [04_승인관리.md](./04_승인관리.md) 참조.

## 현재 한계 / 개선 여지

| 항목 | 현황 | 개선 방향 |
|------|------|----------|
| 용어-단어 자동 매핑 | KOMORAN 분석 결과를 TOKEN_LST에 저장하나 UI에서 활용 미흡 | 용어 등록 시 자동 단어 추출·연결 UI 제공 |
| 코드 도메인 진단 연계 | 코드값 검증 미구현 | 도메인의 ALLOW_VAL_LST와 실제 데이터 비교 진단 추가 |
| 동의어 처리 | ALLOPH_SYNM_LST 저장은 되나 진단 시 부분 활용 | 동의어 기반 유연한 매핑 강화 |
| 일괄등록 검증 | 업로드 시 중복/오류 데이터 처리 기준 불명확 | 업로드 전 유효성 검사 및 오류 리포트 제공 |

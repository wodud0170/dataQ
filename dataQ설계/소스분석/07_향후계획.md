## 중기 개발 과제 (신규 기능)

### 1. 데이터 품질 관리 (DQ) - 최우선 목표

현재 주석 처리된 DQ 기능 구현. 표준화 진단 이후 실제 데이터 품질 측정 단계.

**예상 기능 구성**
```
데이터 품질 관리
├── 품질지표(DQI) 정의    - 어떤 기준으로 품질을 측정할지 정의
├── 핵심관리항목(CTQ)     - 중점 관리할 테이블/컬럼 지정
├── 업무규칙(BR)          - 품질 검증 규칙 정의 (형식, 범위, 참조 등)
├── 검증 실행             - 규칙 기반 실제 데이터 검증 (q-executor)
└── 결과 조회             - 검증 결과 리포트
```

**구현 방향**
- q-executor에서 외부 DB 직접 접속 → 실제 데이터 샘플 조회 → 규칙 적용
- 진단(DiagService) 패턴 재활용 (JOB 관리, 폴링, 중지)

### 2. 수집 스케줄링

```
TB_COLLECT_SCHEDULE 테이블 추가
  - DM_ID, CRON_EXPR, USE_YN
Spring @Scheduled 또는 Quartz Scheduler 적용
수집 완료 후 자동 진단까지 파이프라인 구성
```

### 3. 역할 기반 접근 제어 (RBAC)

```
역할 정의: ADMIN / STANDARD_MANAGER / VIEWER
역할별 메뉴 접근 권한
역할별 API 접근 권한 (Spring Security antMatchers + hasRole)
승인 권한: STANDARD_MANAGER 이상만 가능
```

### 4. 알림 기능 강화

- 승인 요청 시 WebSocket 알림 (현재 인프라 활용)
- 수집/진단 완료 알림
- 이메일 알림 옵션 (Spring Mail)

## 기술 부채

| 항목 | 내용 | 우선순위 |
|------|------|---------|
| Spring Boot 버전 | 2.7.10 → 3.x 업그레이드 필요 (Java 17+ 요구) | 낮음 |
| Vue 버전 | Vue 2.5.2 → Vue 3 마이그레이션 (Vue 2 EOL 2023.12) | 중간 |
| CSRF 비활성화 | 현재 `csrf().disable()` 상태 → 내부망 전용이라면 낮은 위험 | 낮음 |
| API 인증 없음 | 대부분 `permitAll()` → 외부 노출 시 위험 | 중간 |
| WebSocket 연결 안정성 | RabbitMQ STOMP 세션 불안정 이슈 존재 | 중간 |
| 하드코딩 URL | q-executor URL이 설정값이 아닌 코드에 일부 박혀 있을 가능성 | 낮음 |

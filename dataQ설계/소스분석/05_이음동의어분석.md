# 이음동의어(allophSynm) 시스템 분석

## 1. 개요

이음동의어는 하나의 용어/단어에 대한 **동일 의미의 다른 표현**을 관리하는 기능.

```
예: 용어 "기준일자" → 이음동의어: ["기준날짜", "기준일"]
    단어 "기준" → 이음동의어: ["기본", "베이스"]
```

---

## 2. 저장 구조

| 테이블 | 컬럼 | 타입 | 용도 |
|--------|------|------|------|
| TB_TERMS | ALLOPH_SYNM_LST | text[] (PostgreSQL 배열) | 용어 이음동의어 |
| TB_WORD | ALLOPH_SYNM_LST | text[] | 단어 이음동의어 |
| TB_WORD | FORBDN_WORD_LST | text[] | 단어 금칙어 |

- Java ↔ DB 변환: `ArrayTypeHandler` (`com.ndata.common.handler`)
- Java 타입: `String[]`

DSTerm.vue, DSWord.vue, DSCode.vue 모두 동일한 패턴 사용:

- 빈 값은 저장 시 자동 필터링

## 4. 쿼리에서의 활용

### 4-1. UNNEST 패턴 (검색용)

```sql
-- 배열을 행으로 펼침 → 각 이음동의어마다 1행 생성
UNNEST(CASE WHEN array_length(ALLOPH_SYNM_LST, 1) >= 1
  THEN ALLOPH_SYNM_LST
  ELSE '{null}'::text[] END
) as ALLOPH_SYNM
```

**사용 쿼리:**
| 쿼리 | 목적 |
|------|------|
| selectTermsList | 용어 목록 - 이음동의어로도 검색 가능 |
| selectTermsInfoByNm | 용어명/이음동의어 단일 검색 |
| selectCodeInfoList | 코드 목록 - 이음동의어 검색 |
| selectWordList | 단어 목록 - 이음동의어 검색 |

### 4-2. any() 패턴 (직접 비교)

```sql
-- UNNEST 없이 배열에서 직접 검색
WHERE A.TERMS_NM = #{termsNm}
   OR #{termsNm} = any(A.ALLOPH_SYNM_LST)
```

**사용 쿼리:**
| 쿼리 | 목적 |
|------|------|
| selectTermsByNm | 용어 중복 확인 (등록 시) |
| selectCodeInfoByNm | 코드명/이음동의어 검색 |

**차이:**
- UNNEST: 목록 조회 시 사용 (1용어 → N행 → DISTINCT)
- any(): 단건 검색 시 사용 (1용어 → 1행, 성능 우수)

## 5. 진단에서의 활용

### 현재: 미활용

```java
// DiagService.java - 영문명으로만 매칭
Map<String, StdTermsVo> termsByEng = new HashMap<>();
for (StdTermsVo t : allTerms) {
    termsByEng.put(t.getTermsEngAbrvNm(), t);  // 키: 영문약어
}

// 컬럼 영문명으로 검색
StdTermsVo termByEng = termsByEng.get(attrNm.trim());
// → 이음동의어는 참조하지 않음
```

### selectAllTermsForDiag 쿼리

```sql
SELECT T.TERMS_NM, T.TERMS_ENG_ABRV_NM, T.DOMAIN_NM,
       T.ALLOPH_SYNM_LST,  -- 배열 통째로 반환 (현재 미사용)
       D.DATA_TYPE, D.DATA_LEN, D.DATA_DECIMAL_LEN
FROM TB_TERMS T LEFT JOIN TB_DOMAIN D ON T.DOMAIN_NM = D.DOMAIN_NM
```

- allophSynmLst는 SELECT하지만 DiagService에서 사용하지 않음
- UNNEST 미사용 → 용어당 1행 (진단 성능 최적화)

### 활용 가능성

```
현재: 컬럼 한글명 "기준날짜" ≠ 용어 한글명 "기준일자" → TERM_KR_NM_MISMATCH
개선: 컬럼 한글명 "기준날짜" ∈ 이음동의어 ["기준날짜"] → 일치로 처리
```

## 6. 진단에 이음동의어를 적용하려면

### 6-1. 적용 포인트

현재 진단에서 이음동의어가 영향을 줄 수 있는 지점은 **한글명 비교** 단계뿐이다.

```
현재 흐름:
  컬럼 영문명 → 용어 매칭 (영문약어 1:1) → 한글명 비교
                                            ↓
                              표준 한글명 ≠ 실제 한글명 → TERM_KR_NM_MISMATCH

이음동의어 적용 시:
  컬럼 영문명 → 용어 매칭 (영문약어 1:1) → 한글명 비교
                                            ↓
                              표준 한글명 ≠ 실제 한글명
                                            ↓
                              이음동의어에 포함? → YES → 일치로 처리
                                                → NO  → TERM_KR_NM_MISMATCH
```

### 6-2. 구현 방안

**DiagService.java 변경 (한글명 비교 부분)**

```java
// 현재
if (!stdTermsNm.equals(actualKr)) {
    // TERM_KR_NM_MISMATCH
}

// 변경
if (!stdTermsNm.equals(actualKr)) {
    // 이음동의어 배열에 실제 한글명이 포함되어 있는지 확인
    String[] synonyms = termByEng.getAllophSynmLst();
    boolean matchedBySynonym = false;
    if (synonyms != null) {
        for (String syn : synonyms) {
            if (syn != null && syn.equals(actualKr)) {
                matchedBySynonym = true;
                break;
            }
        }
    }
    if (!matchedBySynonym) {
        // TERM_KR_NM_MISMATCH
    }
}
```

**변경 범위:**
- DiagService.java: 한글명 비교 로직 수정 (~5줄)
- 쿼리 변경 불필요: selectAllTermsForDiag에서 이미 ALLOPH_SYNM_LST를 SELECT하고 있음
- UI 변경 불필요

### 6-3. 예시

```
용어: "기준일자" (BASS_DT)
  - 표준 한글명: "기준일자"
  - 이음동의어: ["기준날짜", "기준일"]

컬럼: BASS_DT (한글명: "기준날짜")
  - 현재: "기준날짜" ≠ "기준일자" → TERM_KR_NM_MISMATCH ❌
  - 개선: "기준날짜" ∈ ["기준날짜", "기준일"] → 일치 ✅
```

### 6-4. 진단 결과 표시 개선안

이음동의어로 매칭된 경우, 이슈는 아니지만 사용자에게 알려줄 필요가 있을 수 있다.

**옵션 A: 완전 일치 처리** (단순)
- 이음동의어 매칭 시 이슈로 잡지 않음
- 표준 준수율에 반영 안 됨

**옵션 B: 경고 레벨 이슈로 처리** (상세)
- 새 진단유형: `TERM_KR_NM_SYNONYM` (이음동의어 매칭)
- 이슈 건수에는 포함하지 않되, 컬럼 상세에서 "이음동의어로 매칭됨" 표시
- 사용자가 COMMENT를 표준 한글명으로 변경할지 판단 가능

### 6-5. 금칙어 적용 (추가 고려사항)

단어의 금칙어(FORBDN_WORD_LST)도 진단에 활용할 수 있다.

```
단어 "기준" → 금칙어: ["기본"]
컬럼: GIBON_DT (한글명: "기본일자")

진단: "기본"은 "기준"의 금칙어 → 새 이슈: FORBIDDEN_WORD_USED
```

단, 이 기능은 **컬럼 영문명을 단어 단위로 분리**해야 하므로 현재 진단 구조에서는 복잡도가 높다.
현재 진단은 영문명 전체를 용어와 매칭하는 방식이라, 단어 단위 분석은 별도 설계가 필요.

## 7. 금칙어(forbdnWordLst)

단어(TB_WORD)에만 존재하는 기능.

```
단어 "기준" → 금칙어: ["기본"]
→ "기본"이라는 단어 사용을 금지
```

- 저장/UI 구조는 이음동의어와 동일
- 현재 진단/검증에서 활용하지 않음

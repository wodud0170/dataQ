# dataQ 잠재적 버그 분석

> 주요 소스 파일을 실제로 읽고 분석한 결과입니다.
> 심각도: [Critical] 즉시 수정 필요, [High] 조기 수정 권장, [Medium] 개선 권장, [Low] 참고

## 1. NullPointerException 가능성

### 1-1. [High] SysInfoController - getDataSourceById NPE
- **파일**: `q-center/.../controller/SysInfoController.java` **라인 181-182**
- **내용**: `selectOne`이 null을 반환할 수 있으나, null 체크 없이 바로 `dsVo.setPwd()` 호출
```java
DataSourceVo dsVo = sqlSessionTemplate.selectOne("sysinfo.selectDataSourceById", dsId);
dsVo.setPwd(securityUtils.decryptStr(dsVo.getPwd()));  // dsVo가 null이면 NPE
```
- **영향**: 존재하지 않는 dsId 전달 시 500 에러
- **수정안**: null 체크 후 처리 또는 Optional 사용

### 1-2. [High] DataModelController - getSchemaList NPE
- **파일**: `q-center/.../controller/DataModelController.java` **라인 210-211**
- **내용**: dataSource가 null일 수 있음
```java
DataSourceVo dataSource = sqlSessionTemplate.selectOne("sysinfo.selectDataSourceById", dataVo.getDataModelDsId());
dataSource.setPwd(securityUtils.decryptStr(dataSource.getPwd()));  // null이면 NPE
```
- **영향**: 데이터소스가 삭제된 모델에서 스키마 조회 시 500 에러

### 1-3. [Medium] StructDiagService - clctInfo NPE
- **파일**: `q-executor/.../service/std/StructDiagService.java` **라인 69-70**
- **내용**: clctInfo가 null일 때 `clctInfo.get("clctEndDt")` 호출
```java
Map<String, Object> clctInfo = sqlSessionTemplate.selectOne("datamodel.selectDataModelClctById", this.clctId);
targetClctDt = clctInfo != null ? (String) clctInfo.get("clctEndDt") : null;  // 여기는 처리됨
```
- **참고**: 이 부분은 정상 처리됨. 다만 targetClctId에 대한 유효성 검증은 없음

### 1-4. [Medium] DataStandardController - analyzeTermsBatch 캐스팅
- **파일**: `q-center/.../controller/DataStandardController.java` **라인 1093-1096**
- **내용**: usageCounts의 Map에서 get한 값이 null일 수 있으나, null 체크는 되어 있음. 다만 `(String) row.get("wordNm")`은 DB에서 null이 오면 NPE 없이 null 할당되므로 안전

### 1-5. [High] CustomAuthenticationProvider - admin NPE
- **파일**: `q-center/.../security/CustomAuthenticationProvider.java` **라인 56-58**
- **내용**: admin이 null일 수 있음 (DB에서 조회 실패 시)
```java
AdminVo admin = sqlSessionTemplate.selectOne("login.getUser", id);
if(securityManager.validatePassword(securityManager.encryptSHA256(password), admin.getPassword())) {
```
- **영향**: admin 계정이 DB에서 삭제된 상태에서 로그인 시도 시 NPE

## 3. 트랜잭션 미처리

### 3-1. [High] DataStandardController - createWord/updateWord/deleteWords 트랜잭션 미사용
- **파일**: `q-center/.../controller/DataStandardController.java`
- **라인**: 106-144 (createWord), 147-168 (updateWord), 171-204 (deleteWords)
- **내용**: `sqlSessionTemplate`을 사용하므로 자동 커밋. 하지만 `saveChangeHistory()`가 별도 호출되어, insert는 성공했지만 이력 저장이 실패하면 데이터 불일치 발생
- **영향**: 변경 이력 누락 가능 (saveChangeHistory 내부에서 try-catch로 무시)
- **참고**: createTerms/updateTerms는 `sqlSessionFactory.openSession()` + 수동 commit/rollback 사용 (정상)

### 3-2. [Medium] ExcelUploadService - 행 단위 commit/rollback
- **파일**: `q-common/.../service/ExcelUploadService.java`
- **라인**: 93 (session.commit()), 96 (session.rollback())
- **내용**: 각 행마다 commit/rollback하므로 대량 업로드 시 성능 저하. 또한 하나의 SqlSession을 재사용하면서 commit/rollback을 반복하는 패턴
- **영향**: 성능 이슈 (수천 건 업로드 시 느림), 하지만 데이터 무결성은 유지

### 3-3. [Medium] DataStandardController - registerTermsBatch 이력 저장
- **파일**: `q-center/.../controller/DataStandardController.java` **라인 1496-1521**
- **내용**: 일괄 등록 이력 저장이 `sqlSessionTemplate` (자동 커밋)으로 처리되는데, changehistory.insertChangeHistory와 changehistory.insertChangeHistoryDetail이 각각 독립 커밋되어 마스터는 있지만 상세가 없는 상태 가능
- **영향**: 변경 이력 상세 일부 누락 가능

## 4. 동시성 이슈

### 4-1. [High] DataStandardController - 용어 중복 체크 Race Condition
- **파일**: `q-center/.../controller/DataStandardController.java` **라인 289-291**
- **내용**: 중복 체크(selectTermsByNm)와 등록(insertTerms) 사이에 시간 간격 존재
```java
if (session.selectOne("terms.selectTermsByNm", dataVo.getTermsNm()) != null) {
    throw new Exception("already registered");
}
session.insert("terms.insertTerms", dataVo);  // 다른 요청이 동시에 등록 가능
```
- **영향**: 두 사용자가 동시에 같은 용어를 등록하면 중복 등록 가능
- **수정안**: DB 유니크 제약조건 추가 또는 SELECT FOR UPDATE 사용

### 4-2. [High] 도메인/단어 중복 체크도 동일 패턴
- **파일**: `q-center/.../controller/DataStandardController.java` **라인 738** (createDomain)
- **파일**: `q-common/.../service/ExcelUploadService.java` **라인 60, 123, 232, 268** (각 upload 메서드)
- **영향**: 동일하게 Race Condition 존재

### 4-3. [Medium] DiagService - stopFlags static Map
- **파일**: `q-executor/.../service/std/DiagService.java` **라인 35**
- **내용**: `static final Map<String, AtomicBoolean> stopFlags`를 `ConcurrentHashMap`으로 사용하여 동시성은 처리되었으나, 서비스가 여러 인스턴스(스케일아웃)일 경우 중지 요청이 전파되지 않음
- **영향**: 다중 인스턴스 환경에서 진단 중지 불가 (현재 단일 인스턴스로 문제 없음)

### 9-1. [Medium] DataModelService - ResultSet 닫기 순서
- **파일**: `q-executor/.../service/std/DataModelService.java` **라인 136-137**

- **수정안**: 순서 변경 (rs.close() -> pstmt.close()) 또는 try-with-resources 사용

1. **즉시**: createAdmin/changePassword를 POST로 변경, 데이터소스 비밀번호 마스킹
